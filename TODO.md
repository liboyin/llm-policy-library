**Tooling note — how to run `agy`:** it works, but two things make it look broken.

1. **It hangs unless stdin is closed.** In `-p` mode `agy` blocks on a stdin read (`WCHAN: unix_stream_read_gen`, `fd 0 -> socket`, 0% CPU), and it re-points its own stdout/stderr into `~/.gemini/antigravity-cli/log/cli-*.log`, so whatever it is waiting on is invisible. `--print-timeout` does not bound the run. **Always append `< /dev/null`**; it then answers in ~30s. Note `agy` never "exits 124" — 124 is GNU `timeout`'s own status for "the command timed out and I killed it", not a code from `agy`.
2. **Gemini refuses security-flavoured asks.** "List every bug", "bug hunting", "vulnerability scanning", or "security analysis" on named files are declined outright. Phrase it as an ordinary senior code review and it complies with specific, useful findings.

Scope each call to one module plus its test file; multi-file reviews are slow. `--mode plan` and `--sandbox` both read files without prompting if a read-only guarantee is wanted.

## Phase 3 — Multi-agent system (Microsoft Agent Framework)

Structured messages first — these are the contracts (in `llm_policy_library/models.py`):

- `QueryPlan`: `original_query: str`, `steps: list[PlanStep]` where `PlanStep` = `{search_query: str, purpose: str}` (1–3 steps).
- `RetrievalResult`: `step: PlanStep`, `documents: list[RetrievedDocument]` where `RetrievedDocument` = `{id, title, description, category, score}`.
- `GroundedResponse`: `answer: str`, `citations: list[str]` (control IDs), `is_fallback: bool`.

Agents (each its own module, single responsibility):

- [x] `llm_policy_library/agents/planner.py` — **Planner Agent**: chat agent with structured output (`QueryPlan` JSON schema) and minimized `reasoning_effort` (per D7; no `temperature`/`seed`). Decomposes the user query into 1–3 search steps.
- [x] `llm_policy_library/agents/retrieval.py` — **Retrieval Agent**: no LLM; executes each `PlanStep` against Azure AI Search (hybrid vector+BM25 + semantic reranking when enabled, vector-only otherwise — see the scoring finding below), returns top `RETRIEVAL_TOP_K` (3–5) docs per step, drops results below the mode's relevance floor, deduplicates across steps.
- [x] `llm_policy_library/agents/response.py` — **Response Agent**: chat agent with minimized `reasoning_effort` (per D7; no `temperature`/`seed`); system prompt mandates answering **only** from the provided documents with `[control-id]` citations; if the document set is empty → workflow short-circuits to the safe fallback (`GroundedResponse(is_fallback=True)`) without an LLM call.
- [x] `llm_policy_library/orchestrator.py`: Microsoft Agent Framework **Workflow** (`WorkflowBuilder`) wiring Planner → Retrieval → Response executors with the typed messages above; exposes `async def answer_query(query: str) -> PipelineResult` (plan + retrieved docs + response, for logging/eval). Structured logging at each hop.
- [x] Unit tests: each agent in isolation (mock chat client / mock search client via `patch.object`), orchestration flow with all agents mocked, fallback path, citation extraction. Coverage ≥80% per file.
- [x] Smoke-test live with 1–2 queries once Azure is up; save raw output to `samples/`.

**Status (verified live 2026-07-10):** Phase 3 complete. 161 tests, 100% line/branch coverage on every file; `mypy .` and `ruff check .` clean. Live smoke test over three queries (two on-topic, one off-topic) in `samples/smoke_test.json` + `samples/smoke_test.log`; the off-topic query returns the safe fallback without reaching a chat model. Findings and deviations:

- **`MIN_RELEVANCE_SCORE` is retired, replaced by `MIN_RERANKER_SCORE` (1.8) and `MIN_VECTOR_SCORE` (0.60)** (user decision, 2026-07-10). The carried-forward Phase 1 note assumed the RRF score was merely *mis-scaled*. It is worse: a hybrid `@search.score` is a Reciprocal Rank Fusion score computed from each document's **rank**, never from how well it matches, so some document always scores ~0.03. Measured live: off-topic "What is the capital of France?" scored **0.0323**, tying the best on-topic query. **No threshold on an RRF score can ever fire the safe fallback.** Measured bands (6 on-topic, 4 off-topic questions): reranker 2.00–3.26 vs 0.54–1.44 (separates cleanly); vector-only cosine 0.635–0.776 vs 0.517–0.576 (separates narrowly); hybrid RRF 0.028–0.032 vs 0.024–0.032 (does not separate).
- **Hence: the ranker flag now selects the search *mode*, not just the reranking step** (user decision, 2026-07-10). Ranker on → hybrid + semantic rerank, gate on `@search.rerankerScore`. Ranker off → `search_text` dropped, making it a vector-only search whose `@search.score` is a rescaled cosine, gate on that. The invariant is *the score we rank by is the score we gate on, and it is always calibrated*. Free tier loses BM25; measured cost is small because `content` is prefixed with the control ID, so `AC-2` still retrieves `ac-2` at rank 1.
- **`agent-framework-core` does not ship the chat clients.** As of 1.10 they live in `agent-framework-openai`, and there is **no** `agent-framework-azure-openai` package — the Azure client is `OpenAIChatClient(azure_endpoint=..., api_version=...)`, which targets the **Responses API**. The Phase 1 note claiming "-core includes the Azure OpenAI clients" was wrong; the dependency is now explicit in `pyproject.toml`.
- **`reasoning_effort` is only reachable through the Responses API client.** `OpenAIChatCompletionOptions` has no such key; `OpenAIChatOptions` has `reasoning: {"effort": ...}`. agent-framework's `ReasoningOptions.effort` Literal **omits `"minimal"`**, which is the lowest effort `gpt-5-mini` accepts — it rejects `"none"` with a 400 (verified live; `"none"` is gpt-5.1 only). The value is therefore typed loosely at the call site rather than vendoring an incomplete Literal.
- **Structured outputs work, but constraints are not enforced server-side.** `QueryPlan.steps` deliberately carries no `min_length`/`max_length`: the 1–3 step count is clamped in `plan_query`, so an over-eager model response degrades instead of raising a `ValidationError` deep inside the chat client.
- **The Planner's prompt materially moves retrieval quality.** The first prompt asked for "keywords, topic phrases", and the model emitted keyword soup that scored *worse* on the semantic reranker than the raw question (1.90–2.00 vs 2.12–2.28) — the reranker is trained on natural language. Rewritten to ask for a short natural-language phrase: scores rose to 1.99–2.42 and "logging and monitoring" began correctly planning 3 steps instead of 1. It also had to be told the query goes to an Azure AI Search index, not a web search engine — it was emitting `site:nist.gov` operators.
- **`agent_framework` is now pinned to WARNING** alongside `azure` and `httpx` (`logging_setup.py`): it logs ~6 superstep lines per query at INFO, which buried the audit trail exactly as the Azure SDK did in Phase 2. Same fix, same rationale.
- **The pinned embeddings API version moved to `config.py`.** Query vectors and document vectors are only comparable if both come from the same embeddings API contract, so ingestion and serving now read one constant rather than two that can drift.
- **`pytest-asyncio` added** (`asyncio_mode = "auto"`), since the agents and workflow are async throughout.
